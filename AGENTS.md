# Redline Tech — правила работы агента

## Проект

- Проект: `redline-tech`
- Язык: Java 25
- Minecraft: 1.26.2
- Mod loader: NeoForge 26.2
- Система сборки: Gradle / Gradle Wrapper
- Основная рабочая ветка агента: `gpt`
- `master` является защищённой основной веткой проекта.

Все команды выполнять из корня репозитория `redline-tech`, если явно не указано обратное.

---

## Обязательные правила Git

Перед началом любой задачи:

```powershell
git status
git branch --show-current
git fetch origin
```

Убедиться, что активна ветка:

```text
gpt
```

Если активна другая ветка, не начинать изменения до перехода на `gpt`.

Никогда:

- не изменять `master`;
- не создавать коммиты в `master`;
- не выполнять push в `master`;
- не использовать `git push --force`;
- не использовать `git reset --hard` без прямого указания пользователя;
- не удалять пользовательские незакоммиченные изменения;
- не перезаписывать изменения пользователя;
- не считать локальный commit опубликованным без проверки remote.

Перед изменениями проверить, нет ли существующих пользовательских изменений.

Если рабочее дерево уже содержит изменения, определить их происхождение и не включать несвязанные изменения в текущий патч.

## Навигация по проекту

Использовать MCP `code-review-graph` как основной способ навигации по проекту.

Перед крупной задачей:

```powershell
.\scripts\update-code-graph.ps1
```

Через `code-review-graph` сначала:

1. найти целевые классы и методы;
2. определить прямые зависимости;
3. найти callers и consumers;
4. оценить blast radius;
5. определить затрагиваемые runtime flows.

Только после этого читать конкретные исходные файлы.

Обычные инструменты поиска, включая `rg`, разрешены:

- для проверки конкретной детали;
- для поиска строк, ресурсов и конфигураций;
- если граф не содержит нужных данных;
- если результат графа неоднозначен;
- для проверки результата после изменений.

Не сканировать весь репозиторий обычным поиском, если нужные сведения доступны через `code-review-graph`.

Не доверять графу вслепую.

Учитывать возможный шум от generic method names, например:

```text
build
tick
reload
get
set
apply
```

При неоднозначности использовать fully-qualified symbols, сигнатуры методов и проверять конкретный исходный код.

## Работа с code-review-graph

Граф проекта хранится локально и не является частью исходного кода.

После существенных изменений структуры кода выполнить:

```powershell
.\scripts\update-code-graph.ps1
```

Существенными считаются:

- создание или удаление классов;
- перенос классов между packages;
- изменение основных runtime flows;
- изменение публичных entry points;
- значительное изменение зависимостей между модулями;
- крупный рефакторинг.

После небольших локальных исправлений полная пересборка графа не обязательна.

`.code-review-graph/` не коммитить.

`.code-review-graphignore` является частью конфигурации проекта и может коммититься.

## Принцип внесения изменений

Перед написанием кода:

1. понять существующую архитектуру;
2. найти реальную причину проблемы;
3. определить минимальный корректный blast radius;
4. проверить существующие механизмы, которые можно переиспользовать;
5. только после этого изменять код.

Не исправлять симптомы, если доступна архитектурно корректная причина.

Не создавать параллельную реализацию уже существующего механизма без необходимости.

Не дублировать:

- кеши;
- координатные преобразования;
- worldgen passes;
- storage layers;
- профилировщики;
- конфигурационные параметры.

Предпочитать исправление существующего pipeline созданию дополнительного post-fix хака.

Если post-processing действительно необходим, явно определить:

- фазу выполнения;
- ownership данных;
- порядок относительно Minecraft features;
- поведение на границах чанков;
- повторный запуск;
- memory bounds.

## Правила Worldgen

Старые чанки не обязаны поддерживаться, если пользователь явно не требует обратного.

При изменениях генерации считать основным сценарием генерацию новых чанков.

Worldgen должен быть детерминированным.

Результат генерации не должен зависеть от:

- порядка загрузки чанков;
- направления движения игрока;
- порядка выполнения соседних chunk jobs;
- случайного состояния runtime cache.

Все координатные вычисления должны быть стабильны для одинакового seed и atlas input.

---

## Реки

Речная сеть является связным atlas-guided pipeline.

При изменении рек сохранять следующие инварианты:

- связи HydroRIVERS `NEXT_DOWN` должны сохраняться;
- притоки должны корректно соединяться с downstream segment;
- стык притоков должен иметь согласованный уровень воды;
- профиль воды не должен создавать случайные вертикальные обрывы;
- результат не должен зависеть от порядка загрузки чанков;
- пересекающиеся русла должны корректно объединяться;
- вода реки должна находиться относительно локального берега согласно river profile;
- river water не должна превращаться в lake expansion;
- ложные одиночные водные пятна должны фильтроваться.

Falling blocks русла, включая gravel и sand, обязаны иметь твёрдую поддерживающую подложку.

Не оставлять falling block над:

- воздухом;
- водой;
- заменяемым растением;
- другим нестабильным блоком, если цепочка может обрушиться.

Финальная очистка речного пространства должна учитывать vegetation/features.

Над речной водой и внутри river mask не должны оставаться:

- трава;
- цветы;
- кусты;
- листья;
- стволы;
- другие заменяемые растения.

При необходимости очистки после features учитывать соседние чанки и river boundary ownership.

---

## Озёра и океаны

WorldCover water не должен автоматически превращаться в большое озеро без подтверждения соответствующим water pipeline.

Избегать:

- огромных искусственных песчаных пятен;
- чрезмерно широкого beach halo;
- идеально круглых озёр;
- резкой стены берега;
- одинаковой глубины всех водоёмов.

Берег должен визуально соединяться с окружающим terrain profile.

Изменения береговой логики проверять отдельно для:

- океана;
- большого озера;
- маленького озера;
- реки;
- места соединения реки и другого водоёма.

---

## Структуры и вертикальный мир

Проект использует увеличенный вертикальный диапазон мира(см в датапаке 4096).

Не предполагать ванильную высоту мира.

При работе со структурами, clouds, heightmaps и vertical bounds проверять:

- minimum build height;
- maximum build height;
- surface Y;
- ocean level;
- datapack world height;
- vanilla constants, предполагающие стандартную высоту.

Структуры не должны использовать устаревшую или ванильную surface height, если atlas terrain уже определил фактическую поверхность.

Не оставлять структуры висящими в воздухе из-за несовпадения height pipeline.

---

## Память и кеши

Любой runtime cache должен иметь явный предел роста.

Запрещены безлимитные кеши, зависящие от количества посещённых чанков или atlas samples.

Для каждого нового кеша определить:

- maximum size;
- eviction strategy;
- key representation;
- approximate memory cost;
- thread safety.

Предпочитать primitive storage boxed-коллекциям в горячих worldgen paths.

Избегать массовых временных объектов внутри:

- per-block loops;
- per-column loops;
- biome sampling;
- river sampling;
- GeoTIFF sampling;
- chunk post-processing.

Не увеличивать heap как способ исправления memory leak.

Больший `-Xmx` разрешён для функциональной разработки, но memory-проверки должны выполняться отдельно под ограниченным heap.

При работе с памятью проверять:

- used heap;
- committed heap;
- max heap;
- retained objects;
- cache sizes;
- повторное декодирование atlas tiles.

---

## GeoTIFF и atlas data

Не декодировать один и тот же GeoTIFF tile параллельно несколькими потоками.

Decoded tile cache должен иметь фиксированный предел.

Не хранить глобально все использованные atlas samples.

Не загружать весь GIS dataset в память, если возможны:

- spatial index;
- regional cache;
- tiled access;
- bounded decoded cache.

Изменения atlas cache проверять при длительном перемещении игрока по новой территории.

---

## Потоки и concurrency

Worldgen и atlas processing могут выполняться параллельно.

Не добавлять shared mutable state без анализа thread safety.

Для новых shared structures определить:

- owner;
- readers;
- writers;
- synchronization model;
- lifecycle.

Не использовать глобальную блокировку в горячем worldgen path без измерений.

Не выполнять тяжёлое GIS IO на server tick thread, если это можно подготовить асинхронно.

При исправлении race condition не делать весь pipeline single-threaded без доказанной необходимости.

---

## Производительность

Не оптимизировать на основании ощущений.

Перед крупной оптимизацией определить измеряемую проблему.

Использовать:

- project profiler counters;
- `/rla profile`;
- heap statistics;
- JFR, когда доступен;
- allocation profiling;
- CPU profiling.

Разделять проблемы:

```text
CPU
allocation rate
retained heap
IO
lock contention
chunk scheduling
vanilla height-sensitive algorithms
```

Не смешивать несколько независимых оптимизаций в один патч без необходимости.

При оптимизации ванильного Minecraft через mixin:

1. найти конкретный hot path;
2. подтвердить его профилировщиком;
3. изучить vanilla implementation;
4. оценить влияние NeoForge;
5. сделать минимальный mixin;
6. проверить vanilla-equivalent behaviour;
7. измерить результат до и после.

Не создавать mixin только потому, что метод выглядит медленным.

## Gradle и сборка

Основная проверка проекта выполняется через Gradle Wrapper.

Не считать прямой `javac` заменой Gradle build.

Не считать успешную IDE-компиляцию полной проверкой проекта.

Не изменять `build.gradle`, `settings.gradle` или Gradle configuration только ради устранения warning без полноценного запуска Gradle.

Учитывать Windows path semantics.

Не собирать `file://` URI вручную из Windows paths.

Для файловых Gradle paths предпочитать:

```groovy
file("path").toURI()
```

или Gradle `layout` API.

## Служебные скрипты

В проекте существуют стандартные PowerShell-скрипты.

Все скрипты запускать из корня `redline-tech`.

### Обновление графа

```powershell
.\scripts\update-code-graph.ps1
```

Использовать:

- перед крупной задачей;
- после существенного архитектурного изменения;
- если MCP показывает устаревшие связи.

Не запускать многократно без необходимости во время мелкого локального исправления.

### Проверка проекта

После реализации изменений и целевых тестов выполнить:

```powershell
.\scripts\verify.ps1
```

Это стандартная полная проверка проекта.

Не заменять её:

- прямым `javac`;
- компиляцией одного модуля;
- IDE build;
- `git diff --check`;
- ручным просмотром файлов.

Если `verify.ps1` завершился с ошибкой:

1. не коммитить изменения;
2. определить причину;
3. исправить её;
4. повторно выполнить полный `verify.ps1`.

### Завершение работы

После успешной проверки использовать:

```powershell
.\scripts\finish-agent-work.ps1 -CommitMessage "Краткое содержательное сообщение коммита"
```

Перед запуском проверить:

```powershell
git status --short
```

Не включать в commit несвязанные пользовательские изменения или временные runtime artifacts.

Скрипт является стандартным способом завершения агентской задачи.

После его выполнения проверить:

```powershell
git rev-parse HEAD
git rev-parse origin/gpt
```

Оба commit hash должны совпадать.

Если push не удался, работа не считается опубликованной.

В локальном режиме готовый commit должен остаться в локальном Git repository. Сообщить пользователю точный commit hash и причину неудачного push.

## Временные файлы

Не коммитить:

- runtime logs;
- Minecraft worlds;
- Gradle caches;
- IDEA metadata;
- generated build outputs;
- JFR recordings;
- heap dumps;
- temporary GIS files;
- git bundles, если пользователь явно не потребовал обратного.

Перед commit проверить `git status --short`.

Если инструмент или тест создал временный файл, удалить только этот созданный текущей задачей файл.

Не удалять неизвестные файлы только потому, что они выглядят временными.

---

## Проверка изменений

Во время реализации разрешены целевые проверки конкретного модуля или класса.

После завершения реализации обязательно:

1. выполнить целевые тесты;
2. выполнить `.\scripts\verify.ps1`;
3. проверить `git diff --check`, если он не выполняется скриптом;
4. проверить `git status --short`;
5. убедиться в отсутствии случайных файлов;
6. при существенном изменении структуры обновить code-review-graph;
7. завершить работу через `finish-agent-work.ps1`.

Для исправления worldgen-багов по возможности проверять несколько независимых сценариев, а не одну координату.

Для крупных milestone или архитектурных изменений создавать или обновлять документ в:

```text
docs/worldgen/
```

Документация должна кратко фиксировать:

- проблему;
- найденную причину;
- выбранную архитектуру исправления;
- изменённые runtime flows;
- memory/performance implications;
- выполненные проверки;
- известные ограничения.

Не создавать отдельный большой документ для каждой тривиальной правки.

## Стиль работы

Не спешить ценой качества.

Сначала анализировать существующую реализацию.

При обнаружении дополнительной проблемы в затрагиваемом коде:

- исправить её, если она непосредственно связана с задачей и исправление безопасно;
- иначе зафиксировать её отдельно и сообщить пользователю.

Не расширять scope задачи бесконтрольно.

Не скрывать:

- непроверенные предположения;
- неудачные тесты;
- невозможность визуальной проверки;
- проблемы с push;
- отсутствие profiling evidence.

Не писать, что задача полностью проверена, если выполнялась только компиляция.

Не писать, что код опубликован, пока remote commit не проверен.

---

## Критерий завершения задачи

Задача считается завершённой только когда:

1. причина проблемы найдена и понята;
2. blast radius оценён;
3. изменения реализованы;
4. целевые проверки пройдены;
5. `verify.ps1` завершён успешно;
6. рабочее дерево проверено;
7. изменения закоммичены в `gpt`;
8. commit опубликован в `origin/gpt`;
9. локальный `HEAD` совпадает с `origin/gpt`;
10. пользователь получил краткий отчёт о реализации и проверках.

Если любой пункт невозможно выполнить, явно указать это в итоговом отчёте.
