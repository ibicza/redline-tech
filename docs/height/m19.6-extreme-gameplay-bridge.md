# M19.6 — Extreme height gameplay bridge

M19.6 continues the M19.4/M19.5 true cubic height work.

## Problem

The cube backend can store, save, load, collide and raycast blocks at heights such as:

```text
Y=9000
Y=-12000
```

These positions are inside the real Redline cubic range:

```text
-16384..16383
```

but outside the temporary vanilla `DimensionType` shell:

```text
-2032..2031
```

Vanilla gameplay still uses the shell for several gates:

```text
Entity.checkBelowWorld() kills/discards below level.getMinY() - 64.
ServerGamePacketListenerImpl.handleUseItemOn() rejects placement above/below vanilla build height.
ServerPlayerGameMode block breaking still expects vanilla sections to own the block.
```

## M19.6 behavior

For `redline_world_core:cubic_test` only:

```text
1. Entity.checkBelowWorld no longer kills entities while they are above internalMinY - 64.
2. Normal left-click break packets outside the vanilla shell mutate the cube backend directly.
3. Normal right-click block placement packets outside the vanilla shell mutate the cube backend directly for BlockItem stacks.
4. The edit is saved to Region3D immediately for testing.
5. A native cube-section delta/snapshot is sent back after the edit.
```

Inside the vanilla shell, vanilla interaction remains untouched.

Outside `cubic_test`, vanilla behavior remains untouched.

## Limitations

This is still not the final native renderer.

```text
- Block models outside vanilla shell are still not rendered as normal vanilla chunk meshes.
- Placement uses the BlockItem default block state for now.
- Complex placement states, block orientation, waterlogging and survival break hardness are future work.
- Natural mob spawning outside the vanilla shell still needs a cube-native spawn pipeline.
```
