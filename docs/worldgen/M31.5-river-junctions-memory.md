# M31.5 river junctions, ownership and memory bounds

This patch tightens the M31 river runtime without preserving old generated chunks.

## Fixes

- HydroRIVERS `NEXT_DOWN` is used to orient and connect the river graph.
- Tributaries snap to the downstream segment start when they are within the configured river/bank junction distance.
- Junction profiles use one shared water level and lower the approach smoothly instead of leaving a final one-block cliff.
- Overlapping channel samples prefer the lower/deeper downstream-owned river sample.
- River water level is taken from the cooked atlas profile and clamped below the bank; it no longer depends on loaded terrain from neighbour chunks.
- WorldCover inland-water pixels already claimed by rivers are excluded from lake cooking and lake shore classification.
- Residual WorldCover ponds require a connected component of at least four samples before becoming water.
- Residual WorldCover pond shore radius is capped to 8 blocks to avoid wide random sand patches.
- Water placement replaces replaceable vegetation, grass, flowers and similar surface features.
- Newly loaded river chunks requeue already-loaded neighbouring river boundary chunks for a post-feature cleanup pass.
- Surface polishing resumes unfinished chunks from the saved column cursor when the per-tick column budget is exhausted.
- River fitting uses a fixed primitive sample cache instead of unbounded boxed maps.
- GeoTIFF decoded tile cache is reduced and same-tile duplicate parallel decoding is serialized.
- `/rla profile` reports heap used/committed/max.
- The Gradle publishing repository declaration uses `url = uri(...)` syntax.

## Validation

- `javac` compile of all `redline-atlas-worldgen` Java sources on JDK 25.
- JSON resource parse check with `jq`.
- `git diff --check`.
- Primitive river sample cache stress test: 20 million operations at `-Xmx64m`.
- Synthetic junction graph check: two tributaries connected to one downstream segment.

The attached `problems-report(1).html` was a Gradle Problems Report, not a heap dump. The attached `replay_pid45752.log` showed JIT/runtime references to GeoTIFF and lake/surface river paths but no heap dominator, so the memory changes target the real unbounded and duplicate decoded-array paths found in code.
